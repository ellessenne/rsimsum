## Additional Findings

### 1. High: `char.sep` is internally inconsistent and can break grouping

Files: - `/Users/ellessenne/R-dev/rsimsum/R/utils.R` -
`/Users/ellessenne/R-dev/rsimsum/R/simsum.R` -
`/Users/ellessenne/R-dev/rsimsum/R/multisimsum.R`

Details: - The package documents `char.sep` as a configurable separator
used when splitting grouped data. - However, `.validate_levels()` uses
`grepl(char, levels(...))`, so the separator is treated as a regular
expression rather than a literal string. - This means regex
metacharacters such as `|` are effectively unusable. - Separately,
`.split_by()` still hardcodes `sep = "~"`, so the actual split logic
ignores the configured separator.

Validation: - With `control = list(char.sep = "|")`,
[`simsum()`](https://ellessenne.github.io/rsimsum/reference/simsum.md)
failed validation even though no pipe character was present in the
grouping levels. - With `control = list(char.sep = "_")` and grouping
levels containing `~`,
[`simsum()`](https://ellessenne.github.io/rsimsum/reference/simsum.md)
reached a runtime failure during summary assembly instead of keeping
groups separate.

Why tests miss it: - Current tests only validate `.validate_levels()`
with a few simple literal characters. - I did not find tests that
exercise `char.sep` through
[`simsum()`](https://ellessenne.github.io/rsimsum/reference/simsum.md)
or
[`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md).

### 2. Medium: `summary()` accepts `ci_level = 0` and `ci_level = 1`

Files: - `/Users/ellessenne/R-dev/rsimsum/R/summary.simsum.R` -
`/Users/ellessenne/R-dev/rsimsum/R/summary.multisimsum.R`

Details: - Both summary methods allow boundary values because `ci_level`
is checked with `lower = 0` and `upper = 1`. - These endpoints produce
nonsensical confidence intervals.

Validation: - `summary(x, ci_level = 1)` produced intervals with
`lower = -Inf` and `upper = Inf`. - `summary(x, ci_level = 0)` produced
zero-width intervals equal to the point estimate.

Why tests miss it: - The tests cover values outside the interval, such
as negative numbers or values greater than `1`. - They do not check
boundary values exactly equal to `0` or `1`.

### 3. Medium: invalid `power_df` values are accepted and can silently force `power = 0`

Files: - `/Users/ellessenne/R-dev/rsimsum/R/simsum.R` -
`/Users/ellessenne/R-dev/rsimsum/R/performance.R`

Details: - `control$power_df` is only checked for being numeric. -
Nonpositive values are accepted even though they are not valid degrees
of freedom for [`qt()`](https://rdrr.io/r/stats/TDist.html). - The
resulting `NaN` critical value propagates into the power calculation and
ends up reported as `0`.

Validation: - With `power_df = 0`, `power` was reported as `0`. - With
`power_df = -1`, `power` was also reported as `0`. - The underlying call
to [`qt()`](https://rdrr.io/r/stats/TDist.html) emits `NaN` warnings for
such inputs.

Why tests miss it: - Existing tests only verify that positive values of
`power_df` change the result. - There are no tests for invalid or
nonpositive degrees of freedom.

## Overall Result

- The package test suite passes in the expected execution mode.
- The broader validation pass did not reveal additional failing shipped
  tests.
- It did expose three additional edge-case defects or robustness gaps:
  1.  broken `char.sep` handling
  2.  invalid boundary handling for `ci_level`
  3.  silent acceptance of invalid `power_df`

## Suggested Follow-up

1.  Treat `char.sep` as a literal string in validation and use it
    consistently in splitting logic.
2.  Restrict `ci_level` to the open interval `(0, 1)`.
3.  Require `power_df` to be strictly positive when supplied.
4.  Add regression tests for all three cases.
