# Package Review: Potential Bugs

Date: 2026-03-21

## Findings

### 1. High: `multisimsum()` drops `df` in one valid branch

Files: - `/Users/ellessenne/R-dev/rsimsum/R/multisimsum.R`

Details: - In the branch where `true` is named or `NULL`,
[`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md)
forwards `df` to
[`simsum()`](https://ellessenne.github.io/rsimsum/reference/simsum.md). -
In the branch where `true` is scalar or otherwise non-named,
[`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md)
does not forward `df`. - This means coverage can change depending only
on how `true` is encoded.

Relevant code: - Named branch forwards `df`. - Non-named branch omits
`df`.

Validation: - A minimal reproduction with `df = 1` returned coverage `1`
when `true = c(a = 0, b = 0)`. - The same data returned coverage `0`
when `true = 0`.

Why tests miss it: - Existing tests only exercise named `true` vectors
in `tests/testthat/test-multisimsum.R`. - There is no test for scalar
`true` combined with `df`.

### 2. High: numeric `ci.limits` uses the wrong formula for `becover`

Files: - `/Users/ellessenne/R-dev/rsimsum/R/performance.R`

Details: - The numeric `ci.limits` branch for bias-corrected coverage
uses:

`mean(est) >= est - lower`

instead of checking whether `mean(est)` lies between `lower` and
`upper`.

- The adjacent branches for Wald intervals and column-based limits use
  the correct interval logic.

Validation: - With all estimates equal to `-1` and
`ci.limits = c(-2, 0)`, `cover` returned `1` but `becover` returned
`0`. - The mean estimate is clearly inside the interval, so
`becover = 0` is incorrect.

Why tests miss it: - `tests/testthat/test-ci.limits.R` only checks that
numeric `ci.limits` runs successfully. - It does not validate the
numeric branch against expected values.

### 3. Medium: `rbias` is undefined when `true = 0`, but invalid values are returned silently

Files: - `/Users/ellessenne/R-dev/rsimsum/R/performance.R`

Details: - Relative bias divides by the true value. - When `true = 0`,
the code produces `NaN` or `Inf` for both the estimate and the Monte
Carlo SE. - The package does not guard this case or warn the user.

Validation: - A direct call to `.performance()` with `true = 0` produced
`rbias = NaN` and `mcse = NaN`.

Why tests miss it: - The package includes tests using `true = 0`, but
they only check that execution succeeds or plots are produced. - The
Stata comparison tests explicitly exclude `rbias`.

### 4. Medium: `relprec` and `relerror` break when empirical SE is zero

Files: - `/Users/ellessenne/R-dev/rsimsum/R/performance.R` -
`/Users/ellessenne/R-dev/rsimsum/R/simsum.R`

Details: - `relprec` and `relerror` both divide by `empse`. - If a
method has constant estimates, then `empse = 0` and these statistics
become `Inf` or `NaN`. - The Monte Carlo SE path for `relprec` also
depends on a correlation that is undefined for constant vectors.

Validation: - With constant estimates, `.performance()` returned
`empse = 0`, `relprec = Inf`, `relerror = Inf`, and `relprec` MCSE as
`NaN`.

Why tests miss it: - There do not appear to be tests for zero-variance
estimate vectors.

## Summary

Most of the risk is concentrated in summary-statistic computation: - One
definite forwarding bug in
[`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md). -
One definite formula bug in `becover` for numeric `ci.limits`. - Two
robustness gaps around zero denominators that produce invalid output
without warning.

## Suggested Follow-up

1.  Fix the missing `df` forwarding in
    [`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md).
2.  Correct the numeric `ci.limits` formula for `becover`.
3.  Add regression tests for:
    - scalar `true` with `df` in
      [`multisimsum()`](https://ellessenne.github.io/rsimsum/reference/multisimsum.md)
    - numeric `ci.limits` correctness for `becover`
    - `true = 0` with `rbias`
    - constant-estimate inputs for `relprec` and `relerror`
4.  Decide whether zero-denominator cases should return `NA`, warn, or
    error.
